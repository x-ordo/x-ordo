## 주요 프로젝트 & 고민

### 1. 이혼 전문 변호사용 AI 법률 보조 서비스 (CHAGOK)

* 고민: 법률 영역에서 '자동화'가 어디까지 허용되는가
* 해결: 판단을 대신하지 않고, **정리·구조화·검색**에 집중하는 방향으로 설계

### 2. 콘텐츠 자동 생성 시스템

* 고민: 생성 품질보다 중요한 것은 일관성과 운영 비용
* 해결: 단일 고성능 모델보다, 역할 분리된 멀티 에이전트 구조 채택

### 3. 골프 레슨 플랫폼

* 고민: 기능을 늘릴수록 강사와 수강생 모두 피로해짐
* 해결: 예약·결제·피드백의 최소 흐름만 남기고 나머지는 제거

### 4. 디지털 사이니지 스트리밍 시스템

* 고민: 이론적으로 이상적인 구조 vs 실제 운영 인력의 한계
* 해결: 장애 대응과 로그 추적이 쉬운 구조를 우선 선택

### 5. 대용량 데이터 처리 파이프라인

* 고민: '확장 가능성'이라는 추상적 목표의 함정
* 해결: 현재 데이터 규모와 팀 역량에 맞춘 현실적 설계

### 6. Kotlin + Golang 기반 Windows 마케팅·커머스 애플리케이션

* 개요: 마케팅 자동화 및 커머스 운영용 Windows 데스크톱 앱 (Kotlin UI + Golang 비즈니스 로직)
* 고민 ①: 왜 웹이 아니라 Windows 네이티브인가
  * 고객 환경상 브라우저 사용 불가, 로컬 자원 접근·백그라운드 상주 필수
* 고민 ②: Kotlin + Golang 분리 구조
  * UI 변화 속도 vs 네트워크·병렬 처리 안정성 → IPC 기반 통신 구조 설계
* 해결: "웹보다 빠르고, 서버보다 단순한 구조" 확보

### 7. Windows 온디바이스 LLM 금융 교육 챗봇 플랫폼 (Ollama)

* 개요: 외부 API 의존 없이 Windows 로컬에서 동작하는 금융 교육용 LLM 챗봇
* 고민 ①: 클라우드 없이 LLM을 어떻게 서비스로 만들 것인가
  * GPU 없음, CPU 리소스 제한, 모델 크기·응답 속도·안정성의 삼중 제약
  * 해결: Ollama 기반, 교육 시나리오 중심 프롬프트 설계, 추론 파이프라인 단순화
* 고민 ②: LLM 프로세스 안정성
  * 해결: Python 서비스 + watchdog 구조, 비정상 종료 시 자동 재기동
* 고민 ③: 금융 교육 도메인의 위험성 관리
  * 해결: 응답 톤 제어, 설명형 응답 유도, 특정 질문 유형에서 답변 제한
* 결과: 외부 API 의존도 0, 온디바이스 AI 서비스 구조 실전 경험 확보
